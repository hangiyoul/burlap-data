## 설계 원칙 (FINDINGS §0)
**플랫폼별 얇은 fetch + 업체별 config + LLM 공통 추출 + 표준 스키마.**
업체 50개로 늘려도 코드는 거의 안 늘고, 유지보수는 플랫폼 6개만 본다.

## 구조
```
config/vendors.json   업체 = 설정(platform+base+quirks). 새 업체 = 여기 한 블록
config.py             vendors.json 로더
schema.py             표준 GreenBean (모든 출처 → 한 모양, Cupping Spoon 매핑)
common.py             공통 fetch/HTML 유틸 + 산지 키워드(한글+영문)
adapters/             플랫폼별 입력 어댑터
  cafe24.py   momos, coffeelibre        (2 URL 스킨 + 텍스트스펙)
  godo.py     micoffee/newbean/wbeans/royalcoffee (링크텍스트/alt)
  cobeans.py  cobeans                   (텍스트 풀스펙)
  youngcart.py sewoonggc                (영카트)
  imweb.py    ayantu                    (헤드리스 + 페이지네이션)
  feed.py     gimisa(제휴)              (구글시트/CSV → GreenBean)
extract.py            추출/정규화 + format_name(영문표기) + LLM 자리
pipeline.py           config 기반 fetch→정규화→표준 GreenBean 수집
flavor.py             컵노트 → 맛 카테고리 (Floral/Berry/Chocolate… 13종)
query.py              필터/검색 (국가·지역·가공·품종·컵노트·맛·고도·가격·재고 AND 조합)
watch.py              워치리스트 + 재입고 알림 코어 (품절→판매중 전환 감지)
stock_check.py        재고 판정 (Playwright, 검증: momos 12판매/9품절)
spec_ocr.py           스펙 이미지 추출 (헤드리스+채도선별) + OCR 자리
monitor/snapshot.py   사이트 개편/차단 감지 (수집수 급감 경보)
```

## 필터링 (query.py) — "데이터 내 필터링"
필터 크롤링이 아니라, 넓게 1회 수집 → 표준 GreenBean에서 즉시 조합 필터.
`filter_beans(beans, country=, region=, process=, variety=, cup_note=, flavor=,
altitude_min=, altitude_max=, price_max=, in_stock=)` — 모든 조건 AND.
- **컵노트**: 정확검색(`cup_note="blueberry"`) + 카테고리검색(`flavor="Floral"`, flavor.py 매핑)
- **고도**: min~max 범위 겹침 판정 (`'1,900~2,200m'` 자동 파싱)
- 운영 시 → Supabase DB 쿼리 + 앱 필터 UI(국가▾·가공▾·맛 태그·고도 슬라이더)로 매핑

## 실행
```bash
# 등록 업체 확인
python3 config.py

# 전 업체 카탈로그 → 표준 GreenBean (정적, 설치 불필요)
python3 pipeline.py ethiopia

# 재고/스펙 (헤드리스 — 설치된 Chrome 재사용)
pip3 install --user playwright Pillow
python3 stock_check.py        # momos 에티오피아 생두 재고
python3 spec_ocr.py           # 스펙표 이미지 다운로드

# 모니터링
python3 monitor/snapshot.py save     # 현재 수집수 기준 저장
python3 monitor/snapshot.py check    # 급감 점검
```

## 주요 기능 (앱)
- **검색·필터** — 국가·지역·가공·품종·컵노트·맛카테고리·고도·가격·재고 조합 (query.py)
- **내 업체 저장·편집** — vendors.json(마스터) + 유저 즐겨찾기
- **바로가기** — 커피별 GreenBean.url 딥링크 (판매처로 트래픽 환원)
- **워치리스트 + 재입고 알림** ⭐ — 찜한 커피가 품절→판매중 전환 시 푸시 (watch.py)
  - 전체 아닌 '워치 항목만' 지정 슬롯(≤3회/일) 체크 → 가볍고 점잖음
  - 리텐션 후크 + Pro 전환 명분. 재고 정확도(검증됨)가 곧 알림 정확도

## 해자 (왜 끝장판)
1. **Cupping Spoon 생태계** — 발견→구매→커핑평가→기록이 한 데이터로 (경쟁자엔 커핑앱 없음)
2. **누적 시계열** — 가격·재고·입항 이력 축적 (시간이 막아주는 해자)
3. **정규화 비교 데이터** — LLM로 업체 제각각 표기 통일 (단순 링크모음 ≠ 비교)
4. **워치리스트 알림** — 재고 정확도 기반 재입고 알림 = 없으면 안 되는 소싱 비서

## 운영 메모
- **크롤링 주기**: 기본 1회/일(생두는 실시간 상품 아님). 새 크롭 시즌만 2회/일. 적게 시작→필요 시 증가.
- **워치리스트**: 분 단위 자유시간 대신 프리셋 슬롯(예: 9시/13시/18시 중 ≤3개) 권장.

## 글로벌 확장 전략 (합의)
**원칙: 한 시장(한국) 끝판왕 → 검증된 엔진으로 국가별 확장.** (도구가 아니라 데이터 집계라 번역≠글로벌.)

- **Phase 1 — 한국 끝판왕**: 큐레이션 업체(우리 입력) + 유저 추가/편집. active 9곳 + 제휴(gimisa).
- **Phase 2~ — 국가별 확장**: **별도 앱 N개가 아니라 "한 코드베이스 + 지역(Region) 인식"** 권장(솔로 유지보수).
  App Store 현지화 등록(나라별 이름·스크린샷·키워드)으로 타게팅 느낌은 유지. 엔진 동일, vendors 데이터만 교체.
- **각 국가 = 큐레이션 코어 + 유저 추가 (하이브리드)**:
  - 큐레이션 코어 = 주요 업체 정밀 어댑터 (한국처럼)
  - 유저 추가 = 롱테일, 유저가 본인 거래 업체(회사명·링크) 입력
- **2갈래 추출 엔진**:
  - 큐레이션 업체 → 튜닝 어댑터 (정밀·빠름·쌈)
  - 유저 추가 임의 사이트 → **범용 LLM 추출**(플랫폼 무관) + **우아한 실패**(안 되면 딥링크 폴백)
- **유저 추가 업체 테스트/품질**:
  1. 추가 즉시 **미리보기** → 유저가 검수("맞아요/틀려요")
  2. 크롤링 실패 시 **딥링크 폴백**(업체명+링크는 저장)
  3. **신뢰도 점수** 낮으면 확인 요청
  4. 자주 추가되는 플랫폼(예: Shopify) → **전용 어댑터로 승격**
  5. 개발기 **테스트 코퍼스**(다국가·다플랫폼 20~30개)로 성공률 측정·반복
- **다음 국가 선정 기준**: 스페셜티 시장 크기 / Edan의 업체 관계·지식 / Cupping Spoon 유저 밀집도.
- **버전 차이**: 한국판=큐레이션 마켓플레이스(발견·비교) / 글로벌판=개인 소싱 비서(내 업체 모니터·알림, 워치리스트가 핵심).

## 비즈니스 모델 (확정)
**Burlap = 구독 우선 "연결자(connector)".** 마켓플레이스도, 트레이더도 아님.
- 핵심 수익 = **로스터 구독료**. 돈·재고·물류는 절대 안 건드림(소프트웨어 마진).
- **인센티브 정렬**: 수익이 로스터에서 나오니 오직 로스터 편 → 신뢰 → 더 좋은 제품. (판매자 광고/거래수수료가 주수익이면 중립성 깨짐.)
- **구독 = 반복가치(retention)**: 일회성 '발견'이 아니라 **"내 대신 계속 지켜봐주는"**(워치리스트·재입고알림·모니터링·시계열)이 매달 낼 이유.
- **Free/Pro**: Free=기본 검색(후크) / Pro=워치리스트·알림·고급필터(컵노트·고도)·가격이력·무제한 업체·Cupping Spoon 연동·(글로벌)다지역.
- 그린빈 트레이딩/산지 직거래(아래)는 **언젠가의 별도 사업** — 현재 Burlap 범위 밖.

## 산지 직거래 + 수익 모델 (언젠가·별도 사업, 현재 범위 밖)
> ⚠️ 현재 Burlap은 위 "구독 우선 연결자"에 집중. 아래는 미래 옵션 메모(지금 추진 X).

산지 농장·해외 수입원과 제휴(구글시트 피드, gimisa와 동일 메커니즘) → 로스터를 source 가까이 직접 연결.
**산지 관계 = 복제 불가 해자.** 단, 사업 정체성("연결자" vs "수입업자")을 의식할 것.

- **수익화 (가벼운 것부터)**:
  - 로스터 **Pro 구독** (직거래 리스팅·알림·필터 접근) — 쉬움
  - 업체/산지 **리스팅·featured 비용** — 쉬움
  - **리드(연결) 수수료** (문의·연결당) — 중간
  - 거래 **커미션(%)** — 어려움(거래가 플랫폼 밖이면 추적 난망)
  - 풀 마켓플레이스(결제·물류 직접) — 매우 어려움(=수입업, 자본·규제)
- **정직한 난관**: 생두 거래 ≠ 이커머스. MOQ(마대/컨테이너)·샘플→커핑승인→계약·통관·환율·품질분쟁 → 그래서 수입사가 존재. **Burlap은 "발견·연결"까지 현실적**, 실제 거래(결제·물류·통관) 직접 처리는 별개의 무거운 사업.
- **이해상충/신뢰**: Edan이 특정 산지와 직접 제휴 + 플랫폼 운영 → "리스팅 중립성" 의심 가능. **파트너 배지 명시 + 중립 랭킹**으로 투명성 유지.
- **시퀀싱(닭-달걀)**: 양면시장은 수요·공급 둘 다 필요. **한국 로스터 기반 먼저 구축 → 그게 audience가 되어야 산지가 리스팅함.** 로스터 풀 먼저 → 산지직거래 레이어 얹기.
- **권장**: ①연결/발견 레이어로 시작(구독+리스팅+리드) ②거래 직접 처리는 초기 제외 ③투명성 ④로스터 기반 후 산지 얹기 ⑤수요 검증 후 샘플코디·커미션 단계적 확장.

## 기능 로드맵 (아이디어 풀)
> 기준: "매달 낼 만큼 계속 일해주는가 + 자산 가벼운가 + 로스터 편인가" (구독 우선 연결자).

**🥇 최우선 3 — 구독 엔진의 심장**
1. **저장된 "검색" 알림 (Saved-Search Alert)** — 특정 상품이 아니라 *조건*(예: 에티오피아·워시드·플로럴·2000m↑·5만원↓)에 맞는 신상이 뜨면 알림. → "매달 낼 이유"
2. **Cupping Spoon 원탭 "이 생두 커핑하기"** — 생두 상세 → 산지/지역/품종/가공 자동입력된 커핑 세션 열림. 발견→구매→커핑→기록 루프 폐쇄. 역방향(기록→재구매)도. → "못 떠날 이유"(생태계 해자)
3. **가격·입항 시계열** — 가격 추이("작년보다 8% 쌈") + 입항 캘린더("에티오피아 신상 보통 4~5월") → 구매 계획. → "늦게 와도 못 이기는 이유"(시간 해자)

**🥈 강력한 보조**
4. **가성비 스코어** — kg당 가격 ÷ (등급·고도·컵스코어) 정규화 → 숨은 딜 자동 발굴
5. **비교 뷰 + 향미 레이더** — 생두 2~4개 나란히 비교, 컵노트는 Cupping Spoon 레이더 재사용
6. **취향 기반 추천** — 유저 Cupping Spoon 고득점 기록 분석 → 입맛 맞는 신상 추천

**🥉 워크플로우**
7. **자연어 검색** — "에티오피아 워시드 플로럴 2만원대" → LLM이 필터로 변환
8. **소싱 파이프라인** — 생두에 "샘플주문/받음/커핑함/구매결정" 상태 태그
9. **공유 소싱 리스트** — 큐레이션 목록 팀/클라이언트 공유(PDF/링크, Cupping Spoon PDF 재사용)
